# Procurement Timeline Management — Design Specification

วันที่: 11 กรกฎาคม 2569

สถานะ: พร้อมให้ผู้ใช้ตรวจทานก่อนจัดทำแผนพัฒนา

## 1. เป้าหมาย

สร้างเว็บภาษาไทยบนคลาวด์สำหรับวางแผน Timeline งานจัดซื้อจัดจ้าง ผู้ใช้หลายคนเปิดดูและจัดการข้อมูลชุดเดียวกันได้โดยไม่ต้องเข้าสู่ระบบ ผู้ใช้เลือกวันที่ของขั้นตอนแรก แล้วระบบคำนวณวันที่ของขั้นตอนถัดไปจากแม่แบบมาตรฐาน โดยไม่นับวันเสาร์ วันอาทิตย์ และวันหยุดราชการไทยที่ใช้กับหน่วยงาน

ระบบนี้ใช้สำหรับวางแผนกำหนดการเท่านั้น ไม่ติดตามความคืบหน้าหรือผลการดำเนินงานจริง

## 2. ขอบเขตรุ่นแรก

- สร้าง ดู ค้นหา แก้ไข และลบ Timeline โครงการ
- ผู้ใช้ทุกคนที่มี URL เห็นและแก้ไขข้อมูลร่วมกันได้
- เก็บชื่อโครงการ ผู้รับผิดชอบ วงเงิน วันที่เริ่มต้น และหมายเหตุ
- ใช้แม่แบบมาตรฐานหนึ่งชุดจากภาพอ้างอิงหัวข้อ “29 ล้านบาท”
- ขั้นตอนทั้ง 13 ขั้นเรียงต่อกันแบบลำดับเดียว รุ่นแรกไม่มีขั้นตอนที่ทำงานซ้อนกัน
- คำนวณเฉพาะวันทำการราชการไทย
- แก้วันที่ของขั้นตอนใดก็ได้ แล้วคำนวณขั้นตอนหลังจากนั้นใหม่ตามแม่แบบ
- จัดการรายการวันหยุดที่ใช้ในการคำนวณ
- แสดงผลภาษาไทยและวันที่ พ.ศ. ตามเขตเวลา `Asia/Bangkok`
- มีหน้าสำหรับพิมพ์หรือบันทึก PDF ด้วยความสามารถของเบราว์เซอร์

วงเงินเป็นจำนวนเงินบาทตั้งแต่ศูนย์ขึ้นไปและเป็นข้อมูลประกอบโครงการเท่านั้น รุ่นแรกไม่เลือกหรือเปลี่ยนแม่แบบตามวงเงิน

## 3. สิ่งที่ไม่รวมในรุ่นแรก

- บัญชีผู้ใช้ การเข้าสู่ระบบ สิทธิ์ตามบทบาท หรือการระบุตัวผู้แก้ไข
- การติดตามว่างานเสร็จจริงหรือยัง
- การแจ้งเตือนทางอีเมล LINE หรือช่องทางอื่น
- แม่แบบหลายประเภทและหน้าจอแก้แม่แบบ
- การซิงก์วันหยุดอัตโนมัติจากบริการภายนอก
- การสร้างไฟล์ PDF หรือ Excel ฝั่งเซิร์ฟเวอร์
- Workflow อนุมัติหรือประวัติย้อนกลับทุกเวอร์ชัน

## 5. กติกาการคำนวณ

1. วันที่เริ่มต้นโครงการที่ผู้ใช้เลือกคือวันที่ของขั้นตอนที่ 1 โดยตรง
2. วันที่ของขั้นตอนถัดไปคำนวณด้วย `วันที่ขั้นตอนปัจจุบัน + จำนวนวันทำการของขั้นตอนปัจจุบัน`
3. ตัวอย่าง: ขั้นตอนที่ 1 อยู่วันจันทร์และกำหนดไว้ 4 วันทำการ ขั้นตอนที่ 2 จะอยู่วันศุกร์ หากไม่มีวันหยุดคั่นกลาง
4. ไม่นับวันเสาร์ วันอาทิตย์ และวันที่อยู่ในปฏิทินวันหยุดของระบบ
5. วันสิ้นสุดกระบวนการคำนวณจากวันที่ขั้นตอนที่ 13 บวก 7 วันทำการ
6. วันที่ทั้งหมดเก็บเป็นชนิดวันที่ (`DATE`) ไม่ใช่เวลา เพื่อป้องกันวันที่คลาดเคลื่อนจาก timezone
7. ขั้นตอนที่ 9 ซึ่งระบุว่า “เลือกวันใดวันหนึ่ง” จะได้รับวันที่ชั่วคราวจากการคำนวณอัตโนมัติก่อน และผู้ใช้เปลี่ยนวันที่ได้ด้วยกติกาการปรับกำหนดการเดียวกับขั้นตอนอื่น

## 6. การปรับกำหนดการรายขั้นตอน

- ขั้นตอนทุกแถวมีคำสั่ง “แก้ไขวันที่”
- ขั้นตอนที่ 1 ใช้วันที่เริ่มต้นโครงการ การแก้วันที่นี้เท่ากับเปลี่ยนวันเริ่มต้นและคำนวณ Timeline ทั้งหมดใหม่
- การแก้วันที่ขั้นตอนที่ 1 จะล้างการปรับเองทุกขั้นตอน ระบบแสดงคำเตือนและขอคำยืนยันก่อนดำเนินการ
- สำหรับขั้นตอนที่ 2–13 ผู้ใช้เลือกวันที่ใหม่ของขั้นตอนนั้นได้
- วันที่ใหม่ต้องเป็นวันทำการและต้องอยู่หลังวันที่ของขั้นตอนก่อนหน้า
- ระบบคำนวณจำนวนวันทำการจริงระหว่างขั้นตอนก่อนหน้ากับวันที่ใหม่ และแสดงว่าช่วงนั้นถูกปรับจากแม่แบบกี่วันเป็นกี่วัน
- หากระยะใหม่สั้นกว่าค่าแม่แบบ ระบบเตือนว่าเป็นการลดระยะมาตรฐานและขอคำยืนยัน แต่ยังอนุญาตให้บันทึกเพราะระบบนี้เป็นเครื่องมือวางแผน ไม่ใช่ตัวตรวจข้อกำหนดทางกฎหมาย
- ขั้นตอนก่อนวันที่ที่แก้จะคงเดิม
- ขั้นตอนหลังวันที่ที่แก้จะคำนวณใหม่จากวันที่ใหม่ โดยใช้จำนวนวันมาตรฐานของแม่แบบตามลำดับ
- ขั้นตอนที่แก้เองแสดงป้าย “ปรับกำหนดการ”
- การแก้ขั้นตอนหนึ่งจะล้างการปรับเองของขั้นตอนที่อยู่หลังจากนั้น เพราะวันที่เหล่านั้นถูกคำนวณใหม่ตามแม่แบบ ระบบต้องแสดงข้อความยืนยันก่อนบันทึกหากพบการปรับในช่วงที่จะถูกแทนที่
- มีคำสั่ง “คืนค่าตามแม่แบบ” เพื่อยกเลิกการปรับทุกขั้นตอนและคำนวณใหม่จากวันเริ่มต้น
- “วันสิ้นสุดกระบวนการ” เป็น milestone สุดท้ายที่แก้วันที่ได้ เพื่อปรับช่วง 7 วันของขั้นตอนที่ 13 โดยไม่มีขั้นตอนหลังจากนั้นให้คำนวณต่อ

ตัวอย่าง: ขั้นตอนที่ 1 อยู่วันจันทร์และขั้นตอนที่ 2 เดิมอยู่วันศุกร์ตามระยะ 4 วันทำการ หากผู้ใช้เปลี่ยนวันที่ขั้นตอนที่ 2 เป็นวันอังคารของสัปดาห์ถัดไป ระยะจริงของขั้นตอนที่ 1 จะเปลี่ยนจาก 4 เป็น 6 วันทำการ จากนั้นขั้นตอนที่ 3 จะคำนวณต่อจากวันอังคารด้วยระยะมาตรฐาน 1 วัน จึงอยู่วันพุธหากไม่มีวันหยุดคั่นกลาง

## 7. ปฏิทินวันหยุด

- วันเสาร์และวันอาทิตย์เป็นวันหยุดโดยอัตโนมัติ
- ตารางวันหยุดเก็บวันที่ ชื่อวันหยุด และหมายเหตุอ้างอิง
- ห้ามมีวันหยุดวันที่ซ้ำกัน
- ผู้ใช้เพิ่ม แก้ไข และลบวันหยุดราชการหรือวันหยุดพิเศษที่ใช้กับหน่วยงานได้
- รายการวันหยุดของระบบหมายถึงวันหยุดที่มีผลกับหน่วยงานนี้ทั้งหมด หากมีประกาศเฉพาะพื้นที่ ให้เพิ่มเฉพาะเมื่อประกาศนั้นใช้กับหน่วยงาน
- เมื่อแก้รายการวันหยุด ระบบต้องแจ้งว่า Timeline ที่ครอบคลุมวันที่ดังกล่าวอาจเปลี่ยน และให้ผู้ใช้ยืนยันการคำนวณ Timeline ที่ได้รับผลกระทบใหม่
- การเปลี่ยนวันหยุดทำงานสองช่วง: แสดงรายการโครงการที่ได้รับผลกระทบก่อน แล้วจึงบันทึกวันหยุดและคำนวณใหม่หลังผู้ใช้ยืนยัน
- โครงการที่ได้รับผลกระทบคือโครงการซึ่งช่วงตั้งแต่วันที่ขั้นตอนแรกถึงวันสิ้นสุดครอบคลุมวันที่วันหยุดที่เพิ่ม แก้ หรือลบ
- การคำนวณจากวันหยุดที่เปลี่ยนจะรักษาวันที่ซึ่งผู้ใช้ปรับเองไว้ หากวันที่ปรับเองยังเป็นวันทำการและยังอยู่หลังขั้นตอนก่อนหน้า แล้วคำนวณช่วงอัตโนมัติระหว่าง manual anchor แต่ละจุดใหม่
- หากวันเริ่มหรือวันที่ซึ่งปรับเองกลายเป็นวันหยุดหรือขัดกับลำดับใหม่ ระบบไม่เปลี่ยนวันที่นั้นเงียบ ๆ แต่หยุดการคำนวณที่จุดขัดแย้ง ทำเครื่องหมายโครงการว่า “ต้องตรวจสอบกำหนดการ” และให้ผู้ใช้เลือกวันที่ใหม่ จากนั้นจึงคำนวณช่วงหลังต่อ
- แต่ละปีปฏิทินมีสถานะ “ตรวจสอบรายการวันหยุดครบแล้ว” หาก Timeline ครอบคลุมปีที่ยังไม่ยืนยันสถานะนี้ ระบบเตือนชัดเจนว่าอาจคำนวณไม่ครบแต่ไม่บล็อกการบันทึก ผู้ใช้สามารถเพิ่มข้อมูลวันหยุดแล้วคำนวณใหม่ได้

แหล่งตรวจสอบประกาศควรเป็นข้อมูลทางการ เช่น [สำนักเลขาธิการคณะรัฐมนตรี](https://www.soc.go.th/) และ [รัฐบาลไทย](https://www.thaigov.go.th/)

## 8. หน้าจอและการใช้งาน

### 8.1 Dashboard

- ตาราง Timeline ทั้งหมด
- คอลัมน์หลัก: ชื่อโครงการ ผู้รับผิดชอบ วงเงิน วันที่เริ่ม วันที่สิ้นสุด และวันที่แก้ไขล่าสุด
- ค้นหาจากชื่อโครงการหรือผู้รับผิดชอบ
- กรองตามช่วงวันที่แบบ overlap กล่าวคือแสดงโครงการที่ช่วง Timeline ทับกับช่วงวันที่ค้นหา
- ปุ่ม “สร้าง Timeline”
- คำสั่งเปิดรายละเอียด แก้ไข และลบ โดยการลบต้องยืนยัน

### 8.3 รายละเอียด Timeline

- ส่วนหัวแสดงข้อมูลโครงการ
- ตาราง 13 ขั้นตอน แสดงลำดับ ชื่อขั้นตอน วันที่กำหนด และจำนวนวันจากแถวปัจจุบันไปแถวถัดไป โดยแสดงทั้งค่าแม่แบบและค่าที่ปรับเมื่อแตกต่างกัน
- แสดงวันสิ้นสุดกระบวนการ
- แก้วันที่ได้จากแต่ละแถว
- แก้ข้อมูลโครงการ คืนค่าตามแม่แบบ พิมพ์ และลบได้

### 8.4 จัดการวันหยุด

- ตารางวันหยุดเรียงตามวันที่และกรองตามปี
- เพิ่ม แก้ไข และลบวันหยุด
- ยืนยันได้ว่ารายการวันหยุดของปีนั้นตรวจสอบครบแล้ว พร้อมบันทึกหมายเหตุแหล่งอ้างอิง
- แสดงจำนวน Timeline ที่อาจได้รับผลกระทบก่อนยืนยันการคำนวณใหม่

## 9. สถาปัตยกรรม

- เว็บ Responsive ภาษาไทยแบบ full-stack แอปเดียว
- Frontend และ API ใช้ Next.js กับ TypeScript
- ฐานข้อมูล PostgreSQL บนคลาวด์
- ORM และ migration ใช้ Prisma
- การตรวจข้อมูลทั้งฝั่งเว็บและ API ใช้ schema ชุดเดียวกัน
- Schedule engine เป็นโมดูลแยกที่ไม่มี dependency กับ UI หรือฐานข้อมูล เพื่อให้ทดสอบกติกาวันทำการได้โดยตรง
- Deploy ได้กับผู้ให้บริการคลาวด์ที่รองรับ Node.js และ PostgreSQL โดยรับค่าการเชื่อมต่อผ่าน environment variables

## 10. ส่วนประกอบหลัก

1. **Project service** — จัดการข้อมูลโครงการและ optimistic concurrency
2. **Template repository** — เก็บแม่แบบ 13 ขั้นตอนแบบ seed data อ่านอย่างเดียวในรุ่นแรก
3. **Schedule engine** — คำนวณวันทำการ ความต่างของวัน และ Timeline หลังการแก้วันที่
4. **Holiday service** — จัดการวันหยุดและค้นหา Timeline ที่ได้รับผลกระทบ
5. **Dashboard UI** — ค้นหา กรอง และเปิดโครงการ
6. **Timeline editor** — แสดงและปรับวันที่โดยอธิบายผลกระทบก่อนบันทึก
7. **Print view** — จัดรูปแบบรายละเอียดให้เหมาะกับการพิมพ์และ Save as PDF

## 11. แบบจำลองข้อมูล

### Project

- `id` UUID
- `name`
- `owner_name`
- `budget` decimal
- `start_date` date
- `note` nullable text
- `template_id`
- `process_end_date` date
- `schedule_status` ปกติหรือ “ต้องตรวจสอบกำหนดการ”
- `template_version` รุ่นแม่แบบที่คัดลอกมาเมื่อสร้างโครงการ
- `version` integer สำหรับป้องกันการเขียนทับ
- `created_at`, `updated_at`

### Template / TemplateStep

- ชื่อแม่แบบ
- ลำดับขั้นตอน
- ชื่อขั้นตอน
- จำนวนวันทำการมาตรฐาน
- รุ่นของแม่แบบ

### ProjectStep

- โครงการและลำดับขั้นตอน
- สำเนาชื่อขั้นตอนและจำนวนวันมาตรฐาน ณ เวลาสร้างโครงการ เพื่อไม่ให้โครงการเดิมเปลี่ยนตามแม่แบบในอนาคต
- วันที่กำหนดปัจจุบัน
- `is_date_manually_adjusted` ระบุว่าวันที่ของแถวนี้ถูกเลือกเองหรือไม่
- จำนวนวันจริงจากแถวปัจจุบันไปแถวถัดไปเป็นค่าที่คำนวณจากวันที่ ไม่เก็บซ้ำในฐานข้อมูล สำหรับแถวที่ 13 ให้คำนวณถึงวันสิ้นสุดกระบวนการ

### Holiday

- วันที่แบบ unique
- ชื่อวันหยุด
- หมายเหตุหรือแหล่งอ้างอิง
- `created_at`, `updated_at`

### HolidayCalendarYear

- ปี ค.ศ. ที่ใช้เก็บข้อมูล โดย UI แสดงเป็น พ.ศ.
- สถานะว่าตรวจสอบรายการวันหยุดของปีนั้นครบแล้วหรือยัง
- หมายเหตุแหล่งอ้างอิงและเวลาที่ตรวจสอบล่าสุด

การสร้างโครงการ การแก้วันที่ และการคำนวณขั้นตอนถัดไปต้องบันทึกใน transaction เดียว เพื่อไม่ให้เหลือ Timeline เพียงบางส่วน

## 12. API ระดับสูง

- `GET /api/projects` ค้นหาและกรอง Timeline
- `POST /api/projects` สร้างโครงการและ Timeline
- `GET /api/projects/:id` อ่านรายละเอียด
- `PATCH /api/projects/:id` แก้ข้อมูลโครงการหรือวันเริ่มต้น
- `PATCH /api/projects/:id/steps/:order` ปรับวันที่ขั้นตอนและคำนวณช่วงหลังใหม่
- `POST /api/projects/:id/reset-schedule` คืนค่าตามแม่แบบ
- `DELETE /api/projects/:id` ลบหลังยืนยัน
- `GET /api/holidays` อ่านวันหยุดตามปี
- `POST /api/holidays`, `PATCH /api/holidays/:id`, `DELETE /api/holidays/:id` จัดการวันหยุดและคำนวณผลกระทบ
- การเปลี่ยนวันหยุดใช้ preview/confirm flow และตรวจ Project version อีกครั้งตอนยืนยัน โครงการที่ถูกแก้หลัง preview จะไม่ถูกเขียนทับและจะแจ้ง conflict ให้ผู้ใช้ทราบ

API ต้องตรวจ validation ซ้ำเสมอ แม้ UI จะป้องกันไว้แล้ว

## 13. การใช้งานพร้อมกันหลายคน

- Project มีเลข `version`
- ทุกคำสั่งแก้ไขส่ง version ที่ผู้ใช้กำลังดูไปด้วย
- หากฐานข้อมูลมี version ใหม่กว่า API ตอบ conflict โดยไม่เขียนทับ
- UI แจ้งว่า “ข้อมูลนี้ถูกแก้ไขจากผู้ใช้อื่น กรุณาโหลดข้อมูลล่าสุด”
- เนื่องจากไม่มีระบบบัญชี จะบอกได้เฉพาะเวลาแก้ไขล่าสุด ไม่สามารถระบุชื่อผู้แก้ได้

## 14. การจัดการข้อผิดพลาด

- แสดงข้อความภาษาไทยเฉพาะจุดสำหรับวันที่ไม่ถูกต้อง ข้อมูลบังคับไม่ครบ และวงเงินไม่ถูกต้อง
- เมื่อเครือข่ายหรือฐานข้อมูลขัดข้อง ไม่ล้างค่าที่ผู้ใช้กรอก และไม่แสดงว่าบันทึกสำเร็จ
- การคำนวณและบันทึกที่ล้มเหลวต้อง rollback ทั้ง transaction
- Error log ฝั่งเซิร์ฟเวอร์ต้องไม่บันทึกข้อมูลลับหรือ connection string
- หน้าไม่พบข้อมูลและข้อผิดพลาดระบบมีทางกลับ Dashboard ชัดเจน

## 15. ความปลอดภัย

- ไม่มีการเข้าสู่ระบบตามขอบเขตที่ยืนยัน ดังนั้นทุกคนที่เข้าถึง URL ได้สามารถอ่านและแก้ข้อมูลได้
- ตั้ง `noindex` เพื่อไม่ให้ Search Engine จัดทำดัชนี แต่ไม่ถือเป็นกลไกควบคุมสิทธิ์
- ใช้ HTTPS, security headers, server-side validation และ rate limiting ขั้นพื้นฐาน
- การลบและการคำนวณทับต้องมีหน้าต่างยืนยัน
- ไม่เก็บข้อมูลส่วนบุคคลนอกเหนือจากชื่อผู้รับผิดชอบที่จำเป็นกับโครงการ
